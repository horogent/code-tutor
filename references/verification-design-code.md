# 编程验证方案设计指南

## 核心原则

**验证必须涉及「未见代码」或「未见需求」**。

用讲过的代码例子验证 = 测试记忆 ≠ 测试编程能力。

真正的编程学习 = 面对**从未见过**的代码也能正确判别、面对**从未见过**的需求也能写出正确代码。

---

## 一、编程概念验证（判别模型）

### 1.1 基础验证：新代码判别

给一段用户没见过的代码，判断属于/不属于某编程概念。

**设计原则**：
- 代码的情境要与教学中不同（不同的业务场景、不同的变量名）
- 代码结构要保持概念的共有属性
- 反例最好是"差一点就满足"的边界代码

**好题示例**（验证"纯函数"概念）：
```python
# 给用户看这段从未见过的代码
cache = {}
def get_user(id):
    if id in cache:
        return cache[id]
    user = db.query(id)  # 假设从数据库查询
    cache[id] = user
    return user
```
> 问：`get_user` 是纯函数吗？为什么？
> （答案：不是——它有副作用（修改了外部的 `cache` 字典），且相同输入可能返回不同结果（取决于 cache 状态））

### 1.2 进阶验证：根据概念写新代码

**设计原则**：
- 需求描述不能跟教学中用过的相同
- 需求应足够简单（10行以内可完成）

**好题示例**（验证"闭包"概念）：
> "写一个闭包函数 `make_multiplier(n)`，返回的函数接收一个参数 x，返回 n * x。然后用 `make_multiplier(5)` 创建函数，计算 5 * 10 的值。预测并验证输出。"

### 1.3 边界验证：模糊代码分析

**好题示例**（验证"异步"概念）：
```python
# 这段代码中，console.log 的输出顺序是什么？为什么？
import asyncio

async def main():
    print("A")
    await asyncio.sleep(0)
    print("B")

print("C")
asyncio.run(main())
print("D")
```
> （验证对事件循环的理解——await 会让出控制权，但同步代码先于异步）

## 二、编程命题验证（联结模型）

### 2.1 基础验证：新输入→写代码输出

给一个未见过的新需求，要求写出完整代码。

**设计原则**：
- 需求场景要新颖但仍在已学模式的适用范围内
- 不要求语法完美——关注结构是否正确
- 最好能让用户实际运行验证

**好题示例**（验证"递归"命题）：
> "写一个递归函数 `sum_digits(n)`，计算正整数 n 的各位数字之和。例如 `sum_digits(1234)` 返回 `1+2+3+4 = 10`。提示：基准条件是 n < 10 时返回 n。"

### 2.2 进阶验证：给代码预测输出 + 逆推

**好题示例**（验证"时间复杂度"命题）：
> "以下代码的时间复杂度是多少？如果输入规模变为10倍，运行时间大约增加多少？"
```python
def mystery(arr):
    result = []
    for i in range(len(arr)):
        for j in range(i, len(arr)):
            result.append(arr[i] + arr[j])
    return result
```

### 2.3 边界验证：适用范围 + 修改需求

**好题示例**（验证"二分查找"命题）：
> "你写的二分查找代码，在以下情况还能正确工作吗？
> 1. 数组有重复元素？
> 2. 数组是空数组？
> 3. 查找的元素不存在？
> 如果不行，需要怎么修改？"

## 三、编程综合验证（判联结合）

### 3.1 全流程验证

给出一个完整的新需求场景，要求：
1. 判别这是什么类型的问题（概念判别）
2. 选择合适的解决方案模式（联结模型选择）
3. 写出完整代码（语义转译）
4. 预测运行结果（现象预测）
5. 自测代码是否正确（验证能力）

**好题示例**：
> "你需要实现一个简单的「任务调度器」：可以添加任务（任务是一个函数）、按添加顺序执行所有任务、执行完后清空任务列表。
>
> 1. 这是什么类型的问题？（提示：任务队列、先进先出）
> 2. 适合用什么数据结构？（队列）
> 3. 写出代码实现
> 4. 添加 3 个打印任务，预测执行顺序
> 5. 运行代码验证"

## 四、验证结果判断标准

| 验证表现 | 判断 | 后续行动 |
|---------|------|---------|
| 正确判别 + 正确编码 + 能预测输出 + 预测正确 | 已建构完整模型 | 进入下一单元 |
| 正确判别 + 正确编码但预测输出错误 | 语义层有偏差 | 回到现象层重验 |
| 正确判别但编码错误 | 语义层薄弱 | 回到递进一（语义学习） |
| 能写出代码但说不出用了什么概念 | 模型偏内隐 | 追加1-2题要求说出概念 |
| 错误判别但编码思路正确 | 概念边界不清 | 回到正反例对比 |
| 错误判别且写不出代码 | 判别模型未建构 | 重新教学，换一组代码实例 |
| 能复现讲过的代码但新需求不会 | 假性学习（只记了材料） | 从概念层重新建构 |

## 五、验证题数量建议

| 能力等级 | 每个概念的验证题 | 每个命题的验证题 | 综合验证 |
|---------|---------------|---------------|---------|
| 弱基础 | 2题（新代码判别） | 2题（新需求写代码） | 不作强制 |
| 中等基础 | 1基础 + 1进阶 | 1基础 + 1进阶 | 1题/3单元 |
| 强基础 | 1基础 + 1边界 | 1进阶 + 1边界 | 1题/单元 |

## 六、验证交互协议（强制）

> ⚠️ 这是 code-tutor 验证流程最重要的约束。违反此协议 = 验证无效，用户没有真正被检验。

### 6.1 答案落盘格式

所有验证答案写入 `verification.md`，使用以下强制格式：

```markdown
## 答案

<details>
<summary>题 N 答案（点击展开）</summary>

...完整的参考答案...

</details>
```

**关键约束**：
- 答案统一放在文件末尾的 `## 答案` 二级标题下，不与题目混排
- 每个答案用**独立的 `<details>` 标签**包裹
- 对话中**不使用 `<details>` 标签**（多数对话渲染器不会折叠，反而直接暴露）

### 6.2 对话中的提问流程

1. **在对话中展示题目**（纯文本，只贴题不贴答案）
2. **告知用户文件位置**："答案已折叠在 `verification.md` 中，请先独立作答后再点开对照"
3. **等待用户回答**——这是最关键的一步。不等 = 没验证
4. **评判**：正确→进入下一单元；错误→定位断裂层（现象/语义/概念），回到对应教学环节
5. **揭晓**：评判完毕后，引导用户打开文件折叠区对照

### 6.3 禁止行为清单

| 禁止行为 | 为什么禁止 |
|---------|-----------|
| 在对话中同时贴出题目和答案 | 用户无需思考就能看到答案，验证作废 |
| 在对话中使用 `<details>` 标签 | 多数聊天 UI 不会折叠 HTML 标签，答案直接暴露 |
| 不等用户作答就给出"你答对了吗？答案是..." | 跳过了用户思考环节，等于没验证 |
| 给出太明显的提示 | 等价于泄露答案 |
| 用户答错后直接给出正确答案然后跳过 | 应该回到教学环节重新建构，而非"哦原来如此"就带过 |

### 6.4 正确流程示例

```
导师: (对话中) 验证题：以下代码中 fork+pipe 的程序会卡住还是正常结束？为什么？
      [贴代码]
      答案已折叠在 learning-materials/.../verification.md 中，请先独立作答。

用户: 会卡住，因为子进程没关fd[1]写端...

导师: 完全正确。你可以打开 verification.md 对照参考答案中的详细解释。
      准备好后说"下一章"进入 Unit 2。
```

### 6.5 用户答错时的处理

```
导师: 你的判断有一个关键遗漏——子进程自己持有的 fd[1] 也是写端。
      还记得管道的那条规则吗？"所有写端关闭后，read 才返回 0"。
      这个"所有"包括子进程自己的。我们回到语义层再验证一次...
      [换一段新的反例代码，重新教学]
```
