# 编程渐构方法论完整定义

## 一、什么是编程渐构

**编程渐构**是将「渐构学习理论」应用于编程领域的教学方法论。

**核心观点**：
- 学编程不是背语法和 API——是建构能应对「未见代码/未见需求」的编程思维模型
- 代码的运行结果 = 现象；术语和语法 = 语义；从现象和语义中抽象出的判别规则 = 概念
- 编程教学的核心任务：帮助学习者打通「现象→语义→概念」三层，而非灌输语法规则

---

## 二、信息推测：编程思维的基本单元

### 2.1 定义
编程中的「信息推测」：给定输入（需求/参数/数据），推测输出（代码/结果/行为）。

### 2.2 编程中的两类推测

| 类型 | 特点 | 编程示例 |
|------|------|---------|
| **经验推测** | 基于过去写过的类似代码直接套用 | 见过`for i in range(10)`，照抄写`for i in range(5)` |
| **模型推测** | 基于通用规律映射出未见情况的代码 | 理解了循环的本质（重复执行代码块，用计数器控制次数），能写出任何循环 |

模型推测的核心能力：
- **压缩能力**：一个循环模型概括所有可能的循环场景
- **泛化能力**：面对从未见过的循环需求，能写出正确的循环代码

### 2.3 编程知识的数学描述

编程知识 = 从「输入空间（问题/需求/参数的外延）」到「输出空间（代码/结果/行为的外延）」的「映射关系」

- **输入空间**：所有可能的输入情况（如"所有需要排序的数组"）
- **输出空间**：所有可能的输出情况（如"排序后的数组"）
- **映射关系**：输入到输出的转换规律（如快排算法的分治策略）

## 三、编程判联模型体系

### 3.1 编程判别模型（概念）

**用途**：判断一段代码/一个问题/一个场景"属于"什么概念

**组成**：
- 输入空间：任意代码片段/问题描述/运行现象
- 输出空间：{属于此概念，不属于此概念}
- 判别规则：是否具有「共有属性（内涵）」？

**编程概念示例**：
| 概念 | 内涵（共有属性） | 正例 | 反例 |
|------|---------|------|------|
| 纯函数 | 1.相同输入永远相同输出 2.无副作用（不修改外部状态） | `(x) => x * 2` | `(x) => { counter++; return x * 2; }` |
| 递归 | 1.函数调用自身 2.存在基准条件（终止条件） 3.每次调用向基准条件靠近 | 阶乘函数 | 无终止条件的自调用 |
| RESTful API | 1.使用HTTP方法表达操作 2.URL表示资源 3.无状态 | `GET /users/1` | `POST /getUserById` |

### 3.2 编程联结模型（命题）

**用途**：从已知的编程概念状态推测另一个编程概念的状态

**组成**：
- 输入空间：概念A的外延（如"输入数据"）
- 输出空间：概念B的外延（如"输出数据"）
- 映射关系：从A到B的转换规律（如算法逻辑）

**编程命题示例**：
| 命题 | 输入→输出 | 映射规律 |
|------|----------|---------|
| 递归求解 | 递归问题描述 → 递归函数代码 | 识别基准条件 + 递归关系 + 组合子结果 |
| 状态管理 | 用户操作序列 → UI状态变化 | action → reducer → new state |
| 数据库查询优化 | SQL查询 → 执行计划 | 索引选择 + JOIN策略 + 统计信息 |

### 3.3 编程判联双修

解决任何编程问题至少需要：
- **2个判别模型**：判别输入属于什么问题类型、判别输出代码是否正确
- **1个联结模型**：从问题类型推导代码实现的映射

> 只会判断问题类型（"这是排序问题"）→ 写不出代码
> 只会写代码（"我会快排"）→ 不知道什么时候用
> 两者都有但不匹配（"用快排排序查找问题"）→ 套错算法

## 四、编程三重映射（现象→语义→概念）

这是编程教学与通用渐构教学的核心区别。

### 4.1 编程三重映射

```
┌──────────────────────────────────────────────┐
│         概念层（Concept Layer）                │
│  可泛化的判别模型 + 联结模型                    │
│  "能判断任意代码是否属于某模式"                  │
│  "能从需求推导出代码实现"                       │
├──────────────────────────────────────────────┤
│         语义层（Semantic Layer）               │
│  术语含义 + 代码片段的含义 + API含义            │
│  "这行代码在做什么" "这个函数是什么意思"          │
│  "这个参数代表什么"                            │
├──────────────────────────────────────────────┤
│         现象层（Phenomenon Layer）             │
│  代码运行结果 + 终端输出 + 报错信息              │
│  程序行为 + 性能指标 + 渲染效果                  │
│  "运行后屏幕上显示什么" "终端输出什么"            │
└──────────────────────────────────────────────┘
```

### 4.2 三层映射的核心规则

1. **现象层是入口**：永远先展示运行结果（现象），再解释为什么（语义），最后归纳是什么（概念）
2. **语义层是桥梁**：将现象（运行结果）和概念（抽象模型）连接起来
3. **概念层是目的**：最终目标是建构可泛化的编程思维模型
4. **三层必须打通**：断裂任何一层，编程能力都会缺失

### 4.3 编程中三层的具体对应

| 编程领域 | 现象层 | 语义层 | 概念层 |
|---------|--------|--------|--------|
| 语言语法 | 代码编译/运行的结果（通过/报错/输出） | 语法规则的含义 | 语言设计思想（如类型安全、内存管理） |
| 算法 | 不同输入下的输出结果和执行时间 | 算法步骤的含义 | 算法设计范式（分治/贪心/DP） |
| 框架 | 页面渲染效果、API响应 | 组件/路由/中间件的含义 | 框架设计理念（响应式/声明式/函数式） |
| 调试 | 报错信息、堆栈追踪 | 报错原因分析 | 调试方法论（二分定位/假设验证） |

## 五、三层递进机制（语义驱动概念建构）

### 5.1 递进触发条件

当教学某概念时所使用的语义（术语/语法/API）为初学者未学过的，自动启动三层递进。

**语义依赖检测**：在教学开始前，列出所有涉及的术语和语法点，逐一判断：
- 用户已掌握 → 跳过
- 用户未掌握 → 标记为「待拆解语义」

### 5.2 三层递进流程

```
递进一：语义学习（现象驱动）
  输入：一段可运行的代码或命令
  过程：运行 → 观察现象 → 理解"这段代码做了什么"
  输出：用户能说清"这段代码运行后会发生什么"

递进二：概念构建（从现象归纳含义）
  输入：多个包含该语义的代码实例（正例+反例）
  过程：观察多组现象 → 归纳共性 → 提炼判别规则
  输出：用户能判断"这段新代码是否属于此概念"

递进三：知识构建（建立联结模型）
  输入：已建构的概念 + 新场景
  过程：从输入概念到输出概念的映射推导
  输出：用户能"给定新需求，写出正确代码"
```

### 5.3 三层递进示例：教「闭包」

**触发**：讲闭包概念，用户不知道"函数作用域"这个语义。

**递进一：语义学习（学「函数作用域」）**
```python
# 代码（现象产生工具）
x = 10

def foo():
    x = 20
    print("函数内部:", x)

foo()
print("函数外部:", x)
```
```
# 现象（运行结果）
函数内部: 20
函数外部: 10
```
> 语义：函数内部和外部是同名但独立的变量——这就是「作用域」

**递进二：概念构建（从现象归纳「闭包」）**
- 正例1：内部函数访问外部变量 → 返回内部函数 → 外部变量"存活"
- 正例2：每次调用外层函数创建"独立的作用域副本"
- 反例1：内部函数不访问外部变量 → 只是普通嵌套函数
- 判别规则：闭包 = 函数 + 其引用的外部变量 → 同时满足①函数嵌套②内层引用外层变量③返回内层函数

**递进三：知识构建（建立联结模型）**
- 输入：需要在函数调用间保持状态的需求 → 输出：闭包实现
- 映射规律：外层函数的局部变量→内层函数"捕获"→变量生命周期延长

## 六、编程知识类型

| 类型 | 描述 | 教学策略 | 示例 |
|------|------|---------|------|
| **语法知识** | 语言语法规则 | 现象→语义→概念 | 变量声明、循环语法、函数定义 |
| **语义知识** | 代码片段的行为含义 | 三层递进（这是重点） | 闭包、异步、指针 |
| **惯用模式** | 最佳实践/设计模式 | 多实例对比+适用范围 | MVC、依赖注入、工厂模式 |
| **工具链知识** | 编译器/构建/调试工具 | 命令→现象→场景映射 | Git、Docker、Webpack |
| **领域知识** | 算法/数据结构/系统设计 | 判联双修+复杂度分析 | 排序算法、数据库设计 |

## 七、关键术语表

| 术语 | 编程教学中的定义 | 类比 |
|------|----------------|------|
| 现象层 | 代码运行后能观察到的所有结果 | 你在驾驶座上看到的仪表盘和窗外景色 |
| 语义层 | 每行代码/每个术语的确定含义 | 说明书告诉你每个按钮是干什么的 |
| 概念层 | 从多个语义中抽象出的通用判别模型 | 你理解"好的驾驶习惯"的内核，能应对任何路况 |
| 语义学习 | 通过运行代码产生现象来理解语义 | 先开车上路感受油门和刹车，别急着背交规 |
| 概念构建 | 从现象归纳含义，形成判别能力 | 开了各种路况后，你自然知道"什么是危险驾驶" |
| 知识构建 | 建立输入→输出的联结模型 | 看到前方路况（输入）→预判该做什么操作（输出） |
| 待拆解语义 | 教学中遇到的初学者未掌握的术语/语法 | 教"漂移"前，学生连"方向盘"怎么打都不知道 |
