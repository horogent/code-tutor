# 上下文管理策略

## 三层信息模型

```
┌─────────────────────────────────────────────┐
│  Layer 1: 知识结构（持久保留在上下文中）          │
│  · 全书依赖图（哪些章依赖哪些章）                │
│  · 已建构概念清单（名称+内涵+正反例标签）         │
│  · 已建构命题清单（输入→输出+映射关系）          │
│  · 学习进度状态                               │
│  体积：极小（<5KB），全书学完也不会膨胀            │
├─────────────────────────────────────────────┤
│  Layer 2: 当前工作区（临时保留）                │
│  · 当前章节/单元的材料原文                      │
│  · 当前正在建构的概念的正反例                    │
│  · 当前会话的互动历史                          │
│  体积：中等，章节学完后即蒸馏到 Layer 1，丢弃原文  │
├─────────────────────────────────────────────┤
│  Layer 3: 外部持久化（写入磁盘）                │
│  · 完整进度文件（`.jiangou_progress.json`）    │
│  · 知识地图快照（`.jiangou_knowledge_map.md`） │
│  · 验证记录（`.jiangou_verification_log.md`） │
│  体积：不受限，用于跨会话恢复                    │
└─────────────────────────────────────────────┘
```

## 核心原则

### 原则1：材料用完即弃，知识结构保留
- 一个章节教完后，**丢弃该章节的原文材料**（正例、反例、描述文字）
- **蒸馏保留**：该章节建构了哪些判别模型（概念名+内涵特征）、哪些联结模型（输入→输出映射）
- 蒸馏后的知识结构体积 < 原文的 5%

### 原则2：一章一章加载，不预载全书
- 即使全书有 500 页，当前上下文中**只保留当前章节的原文**
- 上一章的原文已被蒸馏为知识结构
- 下一章的原文尚未加载
- 需要时（如跨章验证）才按需检索上一章的具体材料

### 原则3：每次会话结束，写盘持久化
- 会话结束前，将知识结构和进度写入 `.jiangou_progress.json`
- 下次会话开始时，从文件恢复知识结构（不需要重新加载已学章节的原文）
- 恢复后快速回顾（3分钟），直接继续

### 原则4：上下文预算意识
- 估算当前上下文用量，如果接近极限：
  - 优先丢弃已学章节的原文材料（已蒸馏到知识结构）
  - 其次压缩互动历史（保留关键的验证结果和陷阱标记）
  - 最后才考虑写盘中断，下次继续
- 一句话：**宁可中断会话写盘，也不丢失知识结构**

## 上下文预算监控

| 优先级 | 操作 | 释放量 | 代价 |
|--------|------|--------|------|
| 1 | 蒸馏当前章节，丢弃原文材料，保留知识结构 | 大 | 无（知识结构已保留） |
| 2 | 压缩互动历史为摘要（保留验证结果和陷阱标记） | 中 | 丢失对话细节 |
| 3 | 写盘当前进度，告知用户"下次继续"，自然结束会话 | 全部释放 | 需下次恢复 |

## 生成模式的上下文管理

生成模式同样适用三层信息模型：

1. **每生成完一个模块，立即蒸馏**：将模块中的概念和命题更新到 Layer 1 知识结构
2. **丢弃该模块的原文材料**：生成的 md 文件已落盘，不需要在上下文中保留
3. **只加载当前模块的材料**：不预载全书的材料原文
4. **跨模块引用**：生成后续模块时，如需引用前面模块的概念，从已蒸馏的知识结构（Layer 1）或已生成的 md 文件中查找

> 一本 500 页的书生成教材，上下文消耗与一本 50 页的书基本一致——因为任何时候上下文里只有「知识结构 + 当前模块的材料」。
