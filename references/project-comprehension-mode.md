# 项目解读模式完整方法论

> 以项目源码为载体的编程教学。**项目 = 知识载体，与书/课程没有本质区别。**
> 唯一差异只在概念提取方式：书从文本中识别概念，项目从代码中识别概念。
> 一旦提取出概念和命题，后续教学流程与其他载体**完全相同**。

---

## 一、载体共性模型

### 1.1 所有载体的统一教学流程

```
所有载体（书/课程/项目/框架）：

  Step 1: 从载体中提取概念和命题（提取方式因载体而异）
    - 书/课程 → 阅读目录+内容，识别概念
    - 项目   → 扫描代码结构，识别模块→模块内的概念
    - 框架   → 分析API+设计，识别模式

  Step 2: 按依赖排序 → 模块 → Unit
    ↓ 以下步骤，所有载体完全相同 ↓

  Step 3: 每个 Unit 三层递进教学（现象→语义→概念）
    - 必须写独立可运行代码/指令
    - 必须展示运行结果（现象）

  Step 4: 每个 Unit 落盘到 learning-materials/[主题]/unit-XX/
    - knowledge-point.md + code/ + verification.md

  Step 5: 模块间协作/输入输出分析
    - 模块间的数据流和控制流
    - 联结模型构建

  Step 6: 知识地图 + 验证
```

### 1.2 不同载体的概念提取方式对比

| 载体类型 | 外部输入 | 内部知识 | 概念提取方式 | 概念来源 |
|:--|:--|:--|:--|:--|
| 书/课程 | 文本+大纲 | 知识体系 | 阅读目录+内容，识别概念 | 作者定义 |
| **项目** | **源码+构建系统** | **模块+数据流** | **扫描代码结构，识别模块→提取概念** | **从代码中归纳** |
| 框架 | 文档+API | 设计模式 | 分析API+设计，识别模式 | 框架设计者定义 |

### 1.3 结构同构映射

```
书              项目
─────────────────────────
全书           整个项目
  ├─ 章          ├─ 模块（目录/核心文件组）
  │  ├─ 节       │  ├─ 概念（从代码中提取的抽象）
  │  └─ 节       │  └─ 概念
  └─ 章          └─ 模块
     └─ ...         └─ ...
```

## 二、Phase P0：项目扫描与模块划分（= M0 变体）

### P0.1：确认输入
> "我将开始分析这个项目。项目=知识载体，我会先从代码中提取知识结构（模块→概念），然后用独立可运行代码帮你逐个建构。"

### P0.2：项目元信息收集

用 Glob/Grep/Bash 快速收集：
1. **语言/框架**：检测主要编程语言、构建系统（CMake/Cargo/package.json/go.mod 等）
2. **目录结构**：顶级目录的职责（src/tests/docs/config 等）
3. **入口点**：main 函数、路由注册、启动脚本位置
4. **依赖清单**：第三方依赖的规模和主要依赖
5. **代码规模**：总文件数、核心模块文件数

### P0.3：划分模块（同构于划分书的章节）

从目录结构和入口点推理模块边界：

```
分析原则：
1. 一个独立目录/文件组 = 可能是一个模块
2. 模块有明确的职责边界（输入什么、输出什么）
3. 模块之间通过接口/函数调用连接
```

### P0.4：提取概念（同构于从章节中识别知识点）

从每个模块的代码中提取核心概念：

```
提取原则：
1. 核心 struct/class/interface → 可能是判别模型（概念）
2. 关键函数签名 → 可能是联结模型（命题）
3. 反复出现的设计模式 → 跨模块概念
4. 特殊的数据结构 → 领域概念
```

### P0.5：规模判定 → 解读策略

| 规模 | 特征 | 策略 |
|------|------|------|
| **小型**（<50文件，<5K行） | 单一职责项目 | 全量：所有模块都深度讲解 |
| **中型**（50-200文件，5K-30K行） | 典型业务项目 | 分层：核心模块全讲 + 辅助模块概述 |
| **大型**（>200文件，>30K行） | 企业级/开源大项目 | 聚焦：选一条核心链路，沿链路深度展开 |

### P0.6：向用户呈现"项目知识结构"

```
📦 《[项目名]》知识结构分析

识别到 [N] 个模块，[M] 个核心概念：

模块依赖图：
  [模块A] ──→ [模块B] ──→ [模块C]
       │                      │
       └──→ [模块D] ←────────┘

学习路径：
  阶段一：模块A — [职责]（理解基础设施）
  阶段二：模块B — [职责]（理解核心逻辑）
  阶段三：模块C+D — [职责]（理解完整链路+模块协作）

学完全部后，你将能够：
  - 理解 [项目名] 的架构设计
  - 知道每个模块的输入/输出/职责
  - 提取出 [N] 个可迁移到同类项目的模式
```

## 三、Phase P1：学情诊断 + 现象激活（= M1 + 运行项目）

### P1.1：学情诊断

同宏观模式 M1：
- 对项目所用语言/框架的熟悉程度
- 开发环境是否就绪
- 学习目标：是看懂即可还是想深入修改

### P1.2：现象激活 — 让项目跑起来

> 项目的"整体现象"是后续所有独立的 unit 教学的前提——用户先看到"这个项目整体在干什么"，再逐概念理解"它为什么能这样干"。

1. **检查运行环境**：
   - 读取 README、Makefile、docker-compose 等
   - 检测本地是否具备运行条件
   - 如果有环境：帮助编译/运行
   - 如果无环境：分析入口代码，推理运行行为

2. **运行关键命令获取现象**（根据项目类型调整）：
   - `./program --help`：看命令行接口
   - `cargo run` / `go run` / `python main.py`：直接运行
   - 运行测试：`cargo test` / `go test` / `pytest`
   - HTTP 服务：启动后用 `curl` 打几个端点
   - 示例输入：项目自带的 example/demo 数据

3. **呈现整体现象**：
   > "这是项目运行时你看到的现象：[运行输出]。接下来，我会把这个项目拆解成模块和概念，为每个概念写独立的可运行代码，让你不仅理解这个项目，还能迁移到其他项目。"

## 四、Phase P2：逐 Unit 教学（核心阶段，= M2-M5）

> **这是与其他载体完全一致的教学阶段。** 每个从项目代码中提取的概念，都作为一个独立的 Unit 进行三层递进教学。

### ⚠️ 核心规则：必须写独立可运行代码

**不能只分析项目已有的源码。** 每个概念都必须创建一段**最小的、可独立运行的代码或终端指令**，让用户脱离项目的复杂性，纯粹观察概念本身的现象。

```
错误做法（纯分析）：
  "看，项目里这个文件第42行用到了事件循环，我们来读这段代码..."

正确做法（写独立代码→现象→回到项目）：
  1. 写一段 20 行的独立代码，纯粹展示"事件循环"是什么
  2. 用户运行 → 看到输出（现象）
  3. 解释代码含义（语义）
  4. 提取事件循环的判别规则（概念）
  5. 回到项目："项目中的 [文件:行号] 用的就是同一个模式，因为 [判别特征匹配]"
```

### 每个 Unit 的标准结构

#### 递进零：语义依赖检查
教学每个概念前，检查所用语义是否已知：
> "在理解 [概念X] 之前，我们需要先确认几个词的含义：[列出涉及的术语/语法]"

如有未知语义 → 启动三层递进 → 写独立可运行代码拆解 → 回到项目。

#### 递进一：语义学习（现象驱动）
1. **写一段最小独立可运行代码或终端指令**
2. **展示运行结果**（现象层）
3. **引导观察**："你看到了什么？"
4. **连接语义与现象**："你看到的这个结果，就是 [术语/概念] 的含义——[一句话含义]"

代码文件放在：
```
learning-materials/[项目名]/[模块]/unit-XX/code/demo_*.cpp
```

#### 递进二：概念构建（从现象归纳含义）
1. **展示 3 个正例代码**（独立可运行，体现概念特征）
2. **展示 2 个反例代码**（不符合概念特征）
3. **引导归纳**："这些代码的共同特点是什么？反例有什么不同？"
4. **给出精确内涵**：判别特征+共有属性
5. **回到项目**：标注项目中哪些代码属于这个概念

#### 递进三：知识构建（建立输入→输出联结）
1. 展示 3 个「输入→输出」的独立代码实例
2. 归纳映射规律
3. **回到项目**：项目中的这个映射关系体现在哪几个模块之间
4. 验证：用未见过的输入推测输出

### Unit 落盘结构（与其他载体完全一致）

```
learning-materials/[项目名]/01-[模块名]/unit-01-[概念名]/
├── knowledge-point.md       ← 完整教学内容
│   ├── 现象层（独立代码的运行结果）
│   ├── 语义层（代码含义 + 项目对应位置）
│   ├── 概念层（判别规则 + 正反例）
│   └── 知识构建（输入→输出映射）
├── code/                    ← 独立可运行代码/指令
│   ├── demo_positive1.cpp   ← 正例1
│   ├── demo_positive2.cpp   ← 正例2
│   ├── demo_positive3.cpp   ← 正例3
│   ├── demo_negative1.cpp   ← 反例1
│   ├── demo_negative2.cpp   ← 反例2
│   └── run.sh               ← 一键编译运行
└── verification.md          ← 未见现象验证题
    ├── 题1（现象层判断）
    ├── 题2（语义层解释）
    └── 题3（概念层判别）

注：所有 demo_*.cpp 都是脱离项目的独立最小代码
    在 knowledge-point.md 中标注项目对应的文件路径+行号
```

### 每个模块完成后

写入 `_模块小结.md`：
```
本模块建构的概念：
  - [概念A]（判别模型）：[一句话判别特征]
  - [概念B]（联结模型）：[输入] → [概念B的变换] → [输出]

概念间关系：
  [概念A] ← (输入) → [概念B] ← (输出) → [概念C]

回到项目中：
  - 概念A 体现在 [文件:行号]
  - 概念B 体现在 [文件:行号]
```

## 五、Phase P3：模块协作分析

> 模块协作分析建立的是**联结模型**——模块之间如何通过输入/输出连接。

### P3.1：每个模块的接口定义

```
模块名：[模块A]
  职责：[一句话]
  
  输入接口：
    - [输入1]：[类型/格式] ← 来自 [哪个模块]
    - [输入2]：[类型/格式] ← 来自 [哪个模块]
  
  输出接口：
    - [输出1]：[类型/格式] → 流向 [哪个模块]
    - [输出2]：[类型/格式] → 流向 [哪个模块]
  
  内部概念：
    - [概念A]：[一句话]
    - [概念B]：[一句话]
```

### P3.2：模块间数据流图

```
                    ┌─────────────┐
   [原始输入] ───→  │   模块 A    │
                    │  (解析/验证) │
                    └──────┬──────┘
                           │ [中间数据结构]
                           ▼
                    ┌─────────────┐
                    │   模块 B    │
                    │  (核心处理) │
                    └──────┬──────┘
                           │ [处理结果]
                           ▼
                    ┌─────────────┐
                    │   模块 C    │
                    │  (输出/持久化)│
                    └──────┬──────┘
                           ▼
                      [最终输出]
```

### P3.3：核心链路逐站追踪

选一条端到端链路，逐站标注输入→输出：

```
完整链路：[HTTP请求] → 路由器 → 控制器 → 服务层 → 数据层 → [HTTP响应]

站1: 路由器
  输入:  HTTP Request (URL + Method + Headers)
  输出:  (Controller, params)
  涉及概念: URL模式匹配、中间件链
  项目文件: router.cpp:42-68

站2: 控制器
  输入:  (Controller, params)
  输出:  业务操作指令
  涉及概念: 请求解析、参数验证
  项目文件: controller/user_controller.cpp:15-40

... (逐站)
```

### P3.4：输出模块协作分析文件

写入 `learning-materials/[项目名]/_模块协作分析.md`

## 七、与三层递进的对应关系

| 编程三重映射 | 在项目解读中的体现 |
|:--|:--|
| **现象层** | (a) 项目整体运行输出 (b) **每个概念的独立最小可运行代码的输出** |
| **语义层** | 独立代码的含义 + 项目源码中对应位置的代码含义 + 模块调用关系 |
| **概念层** | 架构模式 + 项目特有的判别模型 + 模块间的联结模型 + 可跨项目泛化 |

---

## 八、核心规则

1. **项目 = 载体，教学 = 相同**：概念提取方式不同（从代码而非文本），但提取后的教学流程与其他载体完全一致
2. **必须写独立可运行代码**：每个概念都要创建脱离项目的最小可运行代码/指令，不能只分析项目源码
3. **现象优先**：先运行独立代码看现象 → 解释语义 → 提取概念 → 回到项目对应位置
4. **输出统一到 learning-materials/**：目录结构和文件模板与其他载体完全一致
5. **四层递进不跳步**：模块划分→概念教学→验证→模块协作，每层落盘后才能进下一层
6. **逐层蒸馏**：每个 Unit 落盘后，该 Unit 的代码原文从上下文丢弃
7. **按需深入**：不是每个模块都深度展开——用户关心的链路才逐个 Unit 讲
8. **泛化收尾**：最后的概念抽象必须能迁移到至少 2 个同类项目
