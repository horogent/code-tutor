# 项目解读模式完整方法论

> 当用户想看懂一个实际项目、理解代码库架构、或问"这个项目是怎么工作的"时使用。
> 本模式将编程三重映射应用于项目代码分析：现象（运行行为/输出）→ 语义（代码含义）→ 概念（架构模式）。

---

## 核心理念

看懂项目 ≠ 读完每一行代码。真正的理解来自：
- **架构层面**：知道项目分了几层，每层干什么 → 判别模型
- **数据流层面**：知道数据从入口到出口经过哪些变换 → 联结模型
- **概念层面**：能提取出项目特有的抽象，并能迁移到同类项目 → 泛化

---

## Phase P0：项目接收与初步扫描

### P0.1：确认输入
> "我将开始分析这个项目。先做一次快速扫描，了解整体轮廓。"

### P0.2：项目元信息收集
通过 Glob/Grep/Bash 快速收集：
1. **语言/框架**：检测主要编程语言、构建系统（CMake/Cargo/package.json/go.mod 等）
2. **目录结构**：顶级目录的职责（src/tests/docs/config 等）
3. **入口点**：main 函数、路由注册、启动脚本位置
4. **依赖清单**：第三方依赖的规模和主要依赖
5. **代码规模**：总文件数、总行数（了解项目大小，规划解读策略）

### P0.3：规模判定 → 解读策略

| 规模 | 特征 | 策略 |
|------|------|------|
| **小型**（<50文件，<5K行） | 单一职责项目 | 全量解读：逐文件讲解 |
| **中型**（50-200文件，5K-30K行） | 典型业务项目 | 分层解读：架构→模块→关键路径 |
| **大型**（>200文件，>30K行） | 企业级/开源大项目 | 聚焦解读：选用户关心的路径，逐层深入 |

## Phase P1：现象层激活 — 让项目"跑起来"

### P1.1：环境检查
- 检查项目是否有构建/运行说明（README、Makefile、docker-compose 等）
- 检测用户本地是否具备运行环境
- 如果有环境：帮助编译/运行，展示实际输出
- 如果无环境：分析入口代码，推理运行行为

### P1.2：运行关键命令，捕获现象

尝试运行这些命令获取现象（根据项目语言/框架调整）：
- 帮助信息：`./program --help`
- 运行测试：捕获测试输出作为"各个功能的现象"
- 示例输入：项目自带的 example/demo
- 启动服务：如果有 HTTP 服务，`curl` 几个端点看响应

### P1.3：现象层输出
将运行结果（终端输出、HTTP 响应、日志）作为「现象」呈现给用户：
> "这是项目运行时你看到的现象。接下来我们进入语义层，理解这些现象是怎么产生的。"

---

## Phase P2：语义层拆解 — 理解每块代码在说什么

按以下层次**逐层**解读，每层完成后写入文件，蒸馏上下文：

### P2.1：架构层语义（宏观结构）

**产出**：`project-analysis/[项目名]/01-架构概览.md`

内容：
1. **分层结构图**（ASCII art 画出项目的分层/分区）
2. **目录-职责映射**：每个目录/模块一句话职责
3. **核心数据结构**：项目最关键的 3-5 个 struct/class/interface
4. **依赖方向**：模块间的依赖关系（箭头标注）

### P2.2：核心链路语义（数据流追踪）

**产出**：`project-analysis/[项目名]/02-核心链路.md`

选取**一条最重要的端到端链路**（如"一个HTTP请求如何被处理并返回响应"），逐站分析：

1. **入口**：请求/输入从哪里进入系统
2. **中间站**：经过哪些函数/模块，每站做了什么变换
3. **出口**：输出/响应是什么，如何序列化
4. **每站现象**：关键代码段 + 实际运行时的中间输出
5. **语义标注**：每处代码的「一句话含义」

### P2.3：模块深度语义（按需深入）

**产出**：`project-analysis/[项目名]/03-模块解析/`

用户关心的模块，每个模块一个文件：
1. 模块的职责和边界（对接哪些其他模块）
2. 关键函数的输入→输出映射
3. 涉及的未知语义拆解（三层递进：现象→语义→概念）

## Phase P3：概念层提取 — 从项目中学可复用的模式

### P3.1：识别项目中的判别模型

从具体代码中提取出**在这个项目中反复出现的概念**：
> "这个项目定义了几个核心概念：[列出概念+判别特征+在代码中的体现]"

示例——分析 Redis 源码可能提取：
- 「事件循环」：判别特征 = 单线程 + epoll/kqueue + 文件事件回调
- 「SDS 动态字符串」：判别特征 = 长度前缀 + 预分配 + 二进制安全 + 兼容C字符串
- 「命令表」：判别特征 = 命令名字符串 → 命令处理函数的映射表

### P3.2：识别项目中的联结模型

提取概念之间的映射关系：
> "这些概念之间的联结规律是：[映射关系]"

示例：`[命令字符串] → 解析器 → [命令对象] → 执行引擎 → [响应对象] → 序列化 → [RESP协议字节流]`

### P3.3：抽象为通用模型

将项目特有的概念/模式**泛化到同类项目**：
> "这个模式不仅存在于这个项目。你在 [同类项目A]、[同类项目B] 中会看到同样的结构。"

每种泛化都给出：
1. 通用模式名（如"事件驱动架构"、"零拷贝序列化"、"插件式命令分发"）
2. 判别特征
3. 在至少 2 个其他项目中的体现

### P3.4：输出概念地图

**产出**：`project-analysis/[项目名]/_项目概念地图.md`

包含：
1. 判别模型清单（项目特有的 + 通用的）
2. 联结模型清单（数据流/控制流映射链）
3. 跨项目泛化表：哪些模式可以在哪些项目中复用

---

## Phase P4：语义依赖拆解

在解读过程中遇到用户不理解的术语/语法时，自动启动三层递进：

### P4.1：标记待拆解语义
> "在理解这段代码之前，我们需要先确认几个词的含义：[列出待拆解的术语/语法]"

### P4.2：就地拆解
1. **现象**：给一段能运行的最小代码，展示这个术语对应的运行结果
2. **语义**：一句话解释这个术语/语法在干什么
3. **概念**：这个术语的判别规则（什么情况下属于/不属于这个概念）

### P4.3：回到项目上下文
> "现在你理解了这个概念，我们回到项目中——这里用到的正是这个概念，因为 [判别特征匹配 + 在项目中扮演的角色]。"

## 项目解读的输出目录

```
project-analysis/                    ← 与 learning-materials/ 同级
└── [项目名]/
    ├── _项目概念地图.md              ← 判别模型+联结模型+跨项目泛化
    ├── 01-架构概览.md               ← 分层图+目录职责+依赖方向
    ├── 02-核心链路.md               ← 数据流逐站追踪
    ├── 03-模块解析/                 ← 按需深入的模块讲解
    │   ├── [模块A].md
    │   └── [模块B].md
    └── 附录/
        ├── 术语表.md                ← 分析过程中拆解的所有术语
        └── 依赖清单.md              ← 第三方依赖及用途说明
```

---

## 核心规则

1. **四层递进不跳步**：架构→链路→模块→概念，每层落盘后才能进下一层
2. **现象优先**：能跑的代码先跑一遍看输出，不能跑的推理运行行为
3. **逐层蒸馏**：每写完一层，该层的代码原文从上下文丢弃，只保留结构摘要
4. **语义必标注**：关键代码必须配「一行含义」，不假设用户能自己看懂
5. **按需深入**：不是每个文件都讲——用户关心的链路才深度展开
6. **泛化收尾**：最后的抽象必须能迁移到至少 2 个同类项目

---

## 与三层递进的对应关系

| 编程三重映射 | 在项目解读中的体现 |
|:--|:--|
| **现象层** | 项目运行输出、HTTP 响应、日志、测试结果 |
| **语义层** | 每段代码的职责、模块间的调用关系、数据变换 |
| **概念层** | 架构模式、项目特有的抽象概念、可跨项目泛化的设计模式 |
