## 一、你要学会什么

**一句话**：判断一段代码在多个线程并发执行时，行为是否可预测、结果是否正确。

**学完之后你能做到**：
- 区分「线程安全」和「线程不安全」的代码
- 识别竞态条件（race condition）的产生原因
- 使用 `std::mutex` + `std::lock_guard` 保护临界区

---

## 二、推测类型判定

线程安全判断属于**未见情况**——每次线程调度的顺序都不同，代码行为可能每次都不同。不能靠"上次运行没问题"就判断安全，必须通过判别模型（共享数据分析 + 同步机制检查）来判定。

## 四、判联模型拆解

### 概念：「线程安全」

**内涵（共有属性/判别特征）**：
1. 多个线程**并发**执行该代码
2. 访问了**共享的可变数据**（至少一个线程在写）
3. 通过**同步机制**（锁/原子操作/无锁算法）消除了竞态条件
4. 结果**始终可预测**，不受线程调度顺序影响

**判别规则**：
- 无共享数据（线程局部/线程隔离）→ **安全**
- 只读共享 → **安全**
- 有共享可变数据 + 无同步 → **不安全**
- 有共享可变数据 + 有正确同步 → **安全**

### 5.2 语义层：代码在说什么

#### 竞态条件的微观过程

`counter = counter + 1` 看似一行，CPU 执行三步：

```
1. LOAD  counter → 寄存器     (读)
2. ADD   寄存器, 1             (改)
3. STORE 寄存器 → counter      (写)
```

两个线程交错时：

```
T1: LOAD  counter → 42
T2: LOAD  counter → 42        ← 还没等 T1 写回！
T1: ADD → 43
T2: ADD → 43                  ← 从 42 加成 43，丢失了一次更新
T1: STORE → counter = 43
T2: STORE → counter = 43      ← 两个线程各加1次，结果只加了1次
```

#### 关键术语

| 术语 | 一句话含义 | 在代码中的作用 |
|------|----------|-------------|
| `std::mutex` | 互斥锁：同一时刻只允许一个线程进入被保护的代码段 | 保护临界区不被并发访问 |
| `std::lock_guard` | RAII 锁包装器：构造时自动上锁，析构时自动解锁 | 防止忘记解锁（即使异常也会解锁） |
| 临界区 (critical section) | 同一时刻只允许一个线程执行的代码段 | `lock_guard` 作用域内的代码 |
| 竞态条件 (race condition) | 多个线程竞争访问同一数据，结果取决于调度顺序 | `counter = counter + 1` 没有锁保护时 |
| RAII | 资源获取即初始化：用对象的生命周期管理资源 | `lock_guard` 离开作用域自动释放锁 |

### 5.3 概念层：抽象出判别模型

<details>
<summary>点击查看精确判别规则</summary>

**判断某段代码是否「线程安全」的规则**：

1. **检查共享数据**：多个线程是否访问同一个变量/对象/内存区域？
   - 否 → 线程安全（没有共享就没有竞争）
   - 是 → 继续第2步

2. **检查可变性**：至少有一个线程在**修改**共享数据？
   - 否（全部只读）→ 线程安全
   - 是 → 继续第3步

3. **检查同步机制**：访问共享可变数据时，是否有正确的同步保护？
   - 锁（`std::mutex`/`std::shared_mutex`）
   - 原子操作（`std::atomic`）
   - 无锁数据结构
   - 是 + 正确使用 → 线程安全
   - 否 → **线程不安全**

</details>

### 5.4 知识构建：从场景到判断

```
输入（场景描述）                        →  输出（安全判断 + 需要的机制）
─────────────────────────────────────────────────────────
多个线程读同一个配置文件                 → 安全，不需要同步
多个线程写同一个日志文件                 → 不安全，需要 std::mutex
每个线程操作各自的 vector               → 安全，数据天然隔离
多个线程读写同一个 hashmap              → 不安全，需要读写锁或分段锁
线程A设置标志，线程B等待标志             → 不安全，至少需要 std::atomic
```

## 七、常见陷阱

1. **以为"看起来不会冲突"就安全**：线程调度的不确定性远超直觉——哪怕一行代码，只要不是原子的，都可能被中断。
2. **锁的粒度太大**：整个函数一把锁 = 多线程变串行 = 性能极差。只锁必要的临界区。
3. **忘记解锁**：使用 `std::lock_guard` 而非手动 `lock()/unlock()`，避免异常路径忘记解锁。
